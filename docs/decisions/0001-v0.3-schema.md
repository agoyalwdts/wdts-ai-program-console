# 0001 — v0.3 schema additions: ExceptionRequest, ReclamationEvent, BudgetSnapshot, FrictionBudgetMetric

**Status:** proposed
**Date:** 2026-04-28
**Authors:** Cursor agent (drafting)
**Decider:** Anuj — needs sign-off before any of this lands in `prisma/schema.prisma`

## Context

The v0.1 schema (in `prisma/schema.prisma`) is an intentional subset of the
canonical schema in `Dashboard_Scoping_v1.md` §3.1. It carries four models —
`User`, `License`, `UsageRecord`, `Decision` — enough to render the four v0.1
read-only views. Three things are now blocked on extending it:

1. **F8 Exception flow** (scoping §2 v1.1) needs `ExceptionRequest` to record
   submitted, attested, FinOps-reviewed, approved/denied exceptions, with a
   max-30-day TTL.
2. **F4 Cursor-seat reclamation + the F1 reclamation card** (scoping §2 v1)
   need `ReclamationEvent` to make the dispute window auditable. F4 currently
   surfaces "likely-reclaim" as a `License.flag` string, which is enough for
   the read view but not for the dispute / decision lifecycle.
3. **F1 trend cards + F10 chargeback** (scoping §2 v1) need a stable monthly
   trend table. Right now `app/(dashboard)/health/page.tsx` derives trends
   on every request from `UsageRecord` — fine at 30 users, will not survive
   the 100K–1M-rows-per-day target in scoping §3.

`FrictionBudgetMetric` (scoping §17.5) is a fourth model bundled here because
it shares the same "snapshot tables that v0.1 derived from `UsageRecord`"
shape and would otherwise need its own ADR.

The v0.1 schema also has three quiet drifts from §3.1 that this ADR cleans
up while we're touching the file:

- `User.roleTag String` (required) → §3.1 specifies `String?` (nullable).
  Deel's `roleTag` can be empty for ex-employees.
- `User.status String` (free-form) → §3.1 specifies a `UserStatus` enum.
- `License.product / UsageRecord.product` are stringly-typed → §3.1 specifies
  a `Product` enum. Same for `LicenseSource`, `UsageDecision`, `DecisionType`.
- `UsageRecord.dlpLayersHit String[]` is missing entirely; F1 / F2 do nothing
  with it today, but the gateway emits it and the manager-queue will soon.

## Decision

Land **one** Prisma migration that introduces:

1. **Four new models** copied verbatim from scoping §3.1: `ExceptionRequest`,
   `ReclamationEvent`, `BudgetSnapshot`, `FrictionBudgetMetric`, plus the
   nine supporting enums (`ExceptionType`, `ExceptionStatus`,
   `ReclamationTrigger`, `ReclamationAction`, `ReclamationState`).
2. **Five enum lifts** on existing models: `User.status` →
   `UserStatus`; `License.product`, `UsageRecord.product` → `Product`;
   `License.source` → `LicenseSource`; `UsageRecord.decision` →
   `UsageDecision`; `Decision.type` → `DecisionType`.
3. **Three field additions / corrections** on existing models:
   `User.roleTag` becomes `String?`; `User.updatedAt DateTime @updatedAt`;
   `UsageRecord.dlpLayersHit String[]`.
4. **No money-column type change.** §3.1 uses `Decimal`; v0.1 uses `Float`.
   Keep `Float` for v0.3 — switching to `Decimal` is a separate ADR with
   non-trivial implications for the Prisma client surface and serialisation.
   File this as an open follow-up.

The migration is named `0002_v0.3_schema` (the v0.1 baseline is migration
`0001`). The companion seed update extends `prisma/seed.ts` with a small
amount of seed data for the four new models so DB-integration tests can
exercise them; v0.1 seed semantics for the four existing models are
preserved.

## Rationale

- **Single migration.** Multiple smaller migrations would each need their
  own seed-update + integration-test pass; the four new models share
  enough scaffolding that splitting them costs more than it buys.
- **Enum lifts piggyback.** Now is the right time to fix the stringly-typed
  drift — the migration touches the same files anyway, and the type-safety
  win surfaces immediately in `lib/integrations/openai/synthetic.ts`,
  `app/(dashboard)/decisions/page.tsx`, and the F10 chargeback page where
  string equality with `"CURSOR"` etc. is currently load-bearing.
- **No `Decimal` change.** Float-to-Decimal is the kind of churn that
  surfaces in every dashboard pixel that prints `formatUsd(x)`. Worth doing,
  but worth doing _once_, deliberately, with a perf+precision benchmark.
- **`FrictionBudgetMetric` included now**, because the v1.2 dashboard will
  read directly from it for the F11/F12 weekly metrics view, and a separate
  migration would just reopen the same files three weeks later.

## Alternatives considered

| Alternative | Why it lost |
|---|---|
| Defer `BudgetSnapshot` until query latency on `UsageRecord` is provably a problem | F1 already does a per-day group-by of every `UsageRecord` since beginning of month on every page load. At 10x current row count this is the next thing to break, and switching to a snapshot table after the fact requires a backfill job. Cheaper to introduce the table now and let the synthetic gateway populate it deterministically. |
| Break the migration into 4 (one per new model) | Each migration would have to ship its own seed delta + DB-integration test pass. Net cost ~3x for ~zero risk reduction. |
| Skip `FrictionBudgetMetric` | F11/F12 in scoping §2 v1.2 read from it. Holding it back doesn't unblock anything; bundling avoids a near-term re-touch of `prisma/schema.prisma`. |
| Switch `Float` → `Decimal` in the same migration | Touches every code path that does arithmetic on `costUsd` / `capUsdMonth`. Out of scope; deserves its own ADR. |
| Adopt the `Decimal` from §3.1 verbatim and absorb the churn | Same reason. |
| Keep `User.roleTag String` (required) | Already crashed once in the synthetic data path when an "imported but not-yet-roled" user was seeded; making it nullable matches Deel reality. |

## Consequences

**Code that needs to change in the same PR as the migration:**

- `prisma/seed.ts` — extend deterministic seed to cover the four new models;
  switch existing string literals to enum members where the schema demands.
- `prisma/schema.prisma` — the migration target (obvious).
- `lib/integrations/gateway/types.ts` — `UsageRecord.product` becomes
  `Product`; existing string callers in F1/F2/F10 keep working because
  Prisma's enum codegen produces a string-literal type.
- `lib/integrations/openai/synthetic.ts`, `lib/integrations/cursor/synthetic.ts`,
  `lib/integrations/gateway/synthetic.ts` — replace the
  `"CURSOR" | "CHATGPT" | …` string unions with the generated enum.
- `app/(dashboard)/health/page.tsx` — gain a fast path that reads from
  `BudgetSnapshot` when present, falls back to live aggregation otherwise.
  Either path acceptable for v0.3; gives us a future-proofing seam without
  forcing the snapshot job to land in the same PR.
- `lib/integrations/gateway/synthetic.db.test.ts` — add coverage for the
  new aggregates.

**Things this ADR _doesn't_ commit to and that still need a human:**

- The actual UI for F8 (exception submission flow). The schema unblocks
  the data layer; the UI is its own scoping pass.
- The dispute-window job that flips `ReclamationEvent.state` from
  `IN_DISPUTE` → `RESOLVED_*` after the timer expires. Will need either
  a cron handler or Prisma-driven scheduled job; out of scope here.
- The `BudgetSnapshot` materialisation job (writes one row per
  product/sub-tier/month). Skeleton can land later.

**Migration plan:**

1. `prisma migrate dev --name v0.3_schema` on a dev DB to generate the SQL.
2. Inspect the generated SQL — confirm `ALTER TABLE … ALTER COLUMN … TYPE`
   for the enum lifts (Prisma generates a `USING` clause cast that's
   safe for the strings v0.1 already writes).
3. `npm run db:seed` to confirm seed works against the new schema.
4. `npm test` to confirm the DB-integration tests still pass.
5. Open the PR. CI re-runs the same gauntlet against the workflow's
   Postgres service container.
6. Once merged: prod (whenever it lands) gets `prisma migrate deploy`.

## Open follow-ups

These are deferred by this ADR but should be tracked:

- **0003 — money column types** (`Float` → `Decimal`). Scope: every
  schema field that holds USD; every code path that arithmetic-s on it;
  every `formatUsd` call site. Not a v0.3 change.
- **0004 — `BudgetSnapshot` materialisation job.** When and how this
  job runs (cron / on-write trigger / manual). Independent of this ADR.
- **0005 — dispute-window timer for `ReclamationEvent`.** As above.
- The `Decision.beforeState / afterState` columns are `String` in v0.1
  ("JSON-as-string"). §3.1 says `Json`. Lift them to `Json` either in
  this migration (cheap) or in 0003 — agent flag, not blocking.
